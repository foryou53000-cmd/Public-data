# 추가 공공데이터 앱 반영 설계

작성일: 2026-05-12  
기준 산출물: `경북_추가공공데이터_수집전처리_2026-05-12.xlsx`  
최종 raw 폴더: `추가공공데이터_수집_전처리_20260512_110300\raw`

## 1. 설계 원칙

이번 추가 데이터는 앱에서 매 검색마다 원자료 전체를 직접 호출하지 않는다.

이유:

- 전국도시공원은 원자료 18,376건, 국립공원 탐방로는 원본 CSV 910,110행이라 브라우저 실시간 호출·파싱에 부적합하다.
- 표준데이터 API와 파일데이터는 CORS, 응답 지연, 인증키 노출, 트래픽 제한 문제가 있다.
- 심사에서는 “실제 공공데이터를 수집·전처리했다”는 재현 가능성이 중요하므로 raw 파일과 전처리 엑셀을 보존하는 편이 더 안전하다.

따라서 구조는 다음처럼 잡는다.

1. `collect_gyeongbuk_public_data.ps1`로 실제 API/파일 원자료 수집
2. 엑셀에서 자동 전처리 결과 검수
3. 검수 통과 후보만 앱용 정적 데이터 파일로 변환
4. 앱 검색 시 기존 실시간 API 결과와 정적 검수 데이터 병합
5. 팝업에는 추천 이유와 함께 공공데이터 출처·수집일·원자료 필드를 표시

## 2. 데이터 계층

앱에 넣을 데이터는 3계층으로 분리한다.

| 계층 | 용도 | 파일/출처 |
|---|---|---|
| 원자료 | 실제 수집 증빙 | `추가공공데이터_수집_전처리_20260512_110300\raw` |
| 검수자료 | 전처리·제외·중복·카테고리 판단 | `경북_추가공공데이터_수집전처리_2026-05-12.xlsx` |
| 앱자료 | 브라우저에서 바로 쓰는 가벼운 후보 목록 | `data/gyeongbuk_public_places.js`, `data/public_data_sources.js` |

앱자료에는 원자료 전체를 넣지 않고, 최종 통과 후보와 검수필요 후보만 넣는다.

## 3. 앱용 장소 스키마

생성 완료 파일: `data/gyeongbuk_public_places.js`

```js
const ADDITIONAL_PUBLIC_PLACES = [
  {
    id: "S05_RURAL_VILLAGE_001",
    sourceId: "S05_RURAL_VILLAGE",
    sourceName: "전국농어촌체험휴양마을표준데이터",
    sourceUrl: "https://www.data.go.kr/data/15013113/standard.do",
    collectedAt: "2026-05-12",
    title: "구산어촌체험휴양마을",
    addr1: "경상북도 울진군 ...",
    sigungu: "울진군",
    lat: 36.0,
    lng: 129.0,
    cat: "농어촌·생산체험",
    sub: "어촌생활체험",
    sourceType: "ruralVillage",
    status: "추천후보",
    curriculumTags: ["촌락과 도시의 생활 모습", "생산과 소비"],
    evidence: "체험프로그램=갯벌체험, 어업체험, 낚시체험",
    overview: "가족 단위 어촌 생활·생산 활동 체험에 적합",
    visitNote: "운영 여부와 체험 가능 일자는 사전 확인 필요",
    rawId: "구산어촌체험휴양마을",
    scoreBase: 85
  }
];
```

필수 필드:

- `id`: 원천ID + 원자료ID 기반 고유값
- `sourceId`, `sourceName`, `sourceUrl`, `collectedAt`
- `title`, `addr1`, `sigungu`, `lat`, `lng`
- `cat`, `sub`, `sourceType`
- `status`: `추천후보`, `보조자료`, `검수필요`, `제외`
- `curriculumTags`
- `evidence`: 원자료 필드 기반 직접 근거
- `overview`
- `visitNote`
- `scoreBase`

## 4. 새 카테고리 체계

기존 앱 카테고리는 `문화유산`, `자연탐구`, `과학체험`, `생활체험`, `인물역사` 중심이다. 새 데이터까지 반영하면 다음처럼 바꾼다.

| 새 카테고리 | 포함 데이터 | 앱 표시 방식 |
|---|---|---|
| 문화유산 | 국가유산청, 관광지 중 유적·사적, 문화유산형 박물관 | 기존 문화유산 유지·강화 |
| 현충·근현대사 | 국가보훈부, 전쟁·독립·의병·6·25 관련 관광지/박물관 | 5학년 6단원 핵심 탭 |
| 자연·생태 | 생태 관광 정보, 자연형 관광지, 생태·수변 공원 | 기존 자연탐구를 명확화 |
| 산지·탐방로 | 국립공원 탐방로, 산림청 등산로 | 장소 카드가 아니라 코스 카드 |
| 공원·생활환경 | 도시공원 중 교육 근거 있는 공원 | 생활권·지도 활용 단원 |
| 농어촌·생산체험 | 농어촌체험휴양마을 | 체험 프로그램 중심 카드 |
| 지역축제 | 문화축제 표준데이터 | 날짜·전년도 참고 배지 필요 |
| 박물관·전시 | 박물관·미술관 표준데이터 | 전시 주제·운영정보 표시 |
| 과학체험 | 기존 Tour API 과학관·천문대 | 기존 기능 유지 |

UI는 탭이 많아지므로 오른쪽 결과 패널 상단을 가로 스크롤 칩 형태로 유지한다. 모바일에서는 2줄 이상으로 늘어나지 않게 `overflow-x` 스크롤이 적합하다.

## 5. 데이터별 반영 규칙

### S01 한국관광공사 생태 관광 정보

- 수집 결과: 경북 1건
- 카테고리: `자연·생태`
- 활용: 생태·지질·환경 보전 단원 보강
- 주의: 좌표가 없어 Tour API 상세 또는 기존 관광 API와 결합해 보정 필요

### S02 전국도시공원정보표준데이터

- 수집 결과: 경북 1,015건
- 카테고리: `공원·생활환경`
- 기본 제외:
  - 단순 어린이공원
  - 단순 소공원
  - 시설 설명이 없는 작은 생활놀이공간
- 유지 조건:
  - 생태, 수변, 문화, 역사, 현충, 과학, 자연학습, 야외공연, 전시, 체육활동 등 교육적 근거가 있는 경우
- 앱 표기:
  - “생활권·공공시설 탐구” 중심으로 낮은 가중치 부여

### S03 국립공원공단 국립공원 탐방로 공간데이터

- 수집 결과: 원자료 910,110행, 경북 후보 215코스
- 카테고리: `산지·탐방로`
- 필드 활용:
  - 탐방코스
  - 상세구간
  - 난이도
  - 거리
  - 통제 여부
  - 대표 좌표
- 주의:
  - 원본이 좌표점 단위라 앱에는 코스 단위로 묶어 넣는다.
  - 지도에는 대표점만 먼저 표시하고, 실제 경로 선은 추후 GeoJSON 변환 후 반영한다.
  - 통제구간은 추천보다 안전 안내 성격으로 표시한다.

### S04 산림청 산림공간정보 등산로정보

- 수집 결과: 전국 432건, 경북 산명 기반 후보 33건
- 카테고리: `산지·탐방로`
- 필드 활용:
  - 산명
  - 등산로 파일 다운로드 URL
  - 산림청 지도 URL
- 주의:
  - 주소가 없으므로 앱에는 `검수필요_산명기반경북판정`으로 넣는다.
  - 좌표계가 산림청 지도 내부 좌표라 일반 위경도 변환이 필요하다.
  - 가족 등산 계획 자료로는 좋지만, 장소 추천 점수는 국립공원 탐방로보다 낮춘다.

### S05 전국농어촌체험휴양마을표준데이터

- 수집 결과: 경북 129건
- 카테고리: `농어촌·생산체험`
- 강한 연결 단원:
  - 촌락과 도시의 생활 모습
  - 생산과 소비
  - 지역문화와 교류
  - 가족 체험 활동
- 앱 표기:
  - 체험프로그램을 카드 칩으로 표시
  - 먹거리체험은 음식점 추천이 아니라 생산·생활문화 체험으로만 설명

### S06 전국문화축제표준데이터

- 수집 결과: 경북 73건
- 카테고리: `지역축제`
- 표시 상태:
  - 예정/진행: `시기형 추천후보`
  - 올해 종료: `보조자료_종료`
  - 전년도 자료: `보조자료_전년도참고`
- 앱 표기:
  - 축제는 상설 장소가 아니므로 장소 추천과 분리된 “시기형 체험” 카드로 표시
  - 2025년 자료는 “작년 이 시기 참고자료”로 명확히 표시

### S07 전국관광지정보표준데이터

- 수집 결과: 경북 57건
- 카테고리:
  - 자연 설명 중심: `자연·생태`
  - 역사·전승 중심: `현충·근현대사` 또는 `문화유산`
  - 일반 관광지: `관광명소` 성격이나 앱 탭에서는 보조자료로 낮춤
- 주의:
  - 숙박시설정보, 접객시설정보, 상가 정보는 추천 근거로 쓰지 않는다.
  - 장소 소개에 교육 키워드가 없으면 `보조자료`로 낮춘다.

### S08 전국박물관미술관정보표준데이터

- 수집 결과: 경북 59건
- 카테고리: `박물관·전시`
- 강한 연결 조건:
  - 소개 필드에 역사, 유물, 문화, 독립, 의병, 전쟁, 과학, 농업, 신라, 고려, 조선 등이 있는 경우
- 앱 표기:
  - 운영시간, 휴관정보, 관람료는 방문 참고로 표시
  - 단순 미술관은 사회 단원 직접 추천보다 문화 체험 보조자료로 낮춤

## 6. 추천 점수 설계

최종 추천은 `기본 점수 + 교육과정 일치 + 거리 + 원자료 신뢰도 - 위험/오염 패널티`로 산출한다.

### 기본 점수

| sourceType | 기본 |
|---|---:|
| heritage | 90 |
| memorial | 90 |
| ruralVillage | 80 |
| museumArt | 75 |
| ecoTour | 75 |
| nationalParkTrail | 65 |
| touristArea | 55 |
| forestTrail | 55 |
| festival | 50 |
| cityPark | 35 |

### 가산

- 단원 핵심 키워드 직접 일치: +30
- 장소 유형 일치: +20
- 시대 태그 일치: +25
- 같은 시군: +15
- 15km 이내: +10
- 원자료 소개/프로그램/축제내용에 직접 근거 있음: +20

### 감산 또는 제외

- 음식점·숙박·카페·일반 상점: 제외
- 역사 단원에 일반 공원·일반 관광지: 제외 또는 보조자료
- 5학년 6단원에 독립운동·민주주의·6·25·국가수호 근거 없음: 제외
- 고조선/삼국/고려/조선 등 시대 단원에 시대 태그 불일치: 제외
- 축제 종료: 보조자료로 낮춤
- 산림청 등산로 주소 미검증: 검수필요로 낮춤

| 고대·삼국 | 선사, 고조선, 신라, 가야, 고분, 왕릉, 고대 유물 | 조선 서원, 항일의병, 6·25 |
| 고려 | 고려, 불교문화, 고려 사찰, 고려 유물 | 신라 전용, 조선 후기, 근현대 전쟁 |
| 조선 | 서원, 향교, 종택, 고택, 조선 유교문화 | 고조선, 6·25, 일반 관광지 |
| 자연환경·지리 | 산지, 하천, 해안, 생태, 지형, 탐방로, 공원 | 실내 전시만 있는 문화시설 |
| 촌락·경제 | 농어촌체험, 생산, 소비, 시장, 지역 특산 | 단순 경관지, 일반 숙박 |
| 지역문화 | 축제, 전통문화, 박물관, 체험마을 | 날짜 지난 축제를 현재 행사처럼 표시 |

## 8. 화면 설계

### 결과 패널

기존 결과 패널 구조는 유지하되 카드 타입을 추가한다.

- 장소 카드: 문화유산, 현충시설, 관광지, 박물관
- 코스 카드: 국립공원 탐방로, 산림청 등산로
- 체험 카드: 농어촌체험휴양마을
- 시기형 카드: 문화축제
- 생활환경 카드: 도시공원

카드 공통 표시:

- 장소명 또는 코스명
- 카테고리 칩
- 출처 칩
- 거리 또는 시군
- 추천상태: 직접추천 / 보조자료 / 검수필요

카드 타입별 추가 표시:

- 코스 카드: 난이도, 거리, 통제여부
- 체험 카드: 체험프로그램 2~3개
- 축제 카드: 시작일, 종료일, 전년도 참고 여부
- 박물관 카드: 휴관정보, 관람료 요약

### 팝업

팝업에는 다음 순서로 표시한다.

1. 장소 소개
2. AI 교육과정 연결
3. 단원
4. 연계 판단: 직접 연계 / 보조 연계 / 검수 필요
5. 추천 이유
6. 직접 연계 근거
7. 원자료 출처와 수집일
8. 수업 활동
9. 방문 전 확인사항
10. 핵심 키워드

## 9. 구현 순서

1. 엑셀 `02_최종마스터_중복제거` 기준으로 앱용 데이터 파일 생성
   - 예정 파일: `data/gyeongbuk_public_places.js`
   - 제외 후보와 검수필요 후보는 별도 상태로 분리

2. `경북체험학습추천 (2).html` 카테고리 정의 교체
   - `CATS`를 새 카테고리 체계로 변경
   - 소분류도 새 sourceType에 맞게 재정리

3. 기존 추천 수집 로직에 정적 데이터 병합
   - 현재 `doSrc()`는 Tour API, 국가유산청, 국가보훈부를 병합한다.
   - 여기에 `fetchAdditionalPublicCandidates(u, local)`을 추가한다.

4. `buildAnyPlace()` 또는 별도 변환 함수 추가
   - `buildAdditionalPublicPlace(raw, ctr, u, local)`
   - 기존 `place` 객체와 같은 구조로 맞춘다.

5. `scorePlace()` 확장
   - `sourceType`별 기본 점수와 단원 가드 적용
   - `검수필요`는 기본 추천 결과 하단 또는 별도 토글로 표시

6. 카드 렌더링 확장
   - `sourceType==='nationalParkTrail'`
   - `sourceType==='forestTrail'`
   - `sourceType==='festival'`
   - `sourceType==='ruralVillage'`

7. 팝업 근거 문장 개선
   - `evidence` 필드가 있을 때만 직접 연계 근거 생성
   - 근거가 약하면 “보조 연계”로 표시

8. 검증 검색
   - 5학년 6단원: 현충·근현대사 결과가 충분한지 확인
   - 자연환경/지리 단원: 산지·탐방로가 나오는지 확인
   - 촌락/경제 단원: 농어촌체험마을이 나오는지 확인
   - 지역문화 단원: 축제가 날짜 상태와 함께 나오는지 확인
   - 음식점·숙박·카페가 섞이지 않는지 확인

## 11. 우선 적용 범위

1차 적용은 모든 데이터를 무작정 넣지 않는다.

우선 반영:

- 농어촌체험휴양마을
- 박물관·미술관
- 관광지정보 중 역사·자연 근거가 강한 곳
- 국립공원 탐방로 중 대표 좌표와 코스명이 있는 후보
- 문화축제 중 예정/계절성 설명이 가능한 후보

보류 또는 검수필요:

- 교육 근거 약한 도시공원
- 주소 없는 산림청 등산로
- 종료된 축제
- 일반 관광지 중 숙박·상가 중심 설명이 강한 곳

이렇게 가야 장소 수는 늘어나면서도, “교육과정과 안 맞는 추천”과 “음식점·숙박 오염”을 동시에 줄일 수 있다.
